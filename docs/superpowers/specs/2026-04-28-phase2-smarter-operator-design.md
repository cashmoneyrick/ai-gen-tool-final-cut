# Phase 2: Smarter Operator — Design Spec

**Date:** 2026-04-28
**Status:** Approved

---

## Problem

The operator currently has no ability to evaluate generated images. It relies entirely on Rick's numeric ratings and whatever verbal notes he gives. When the same failure repeats, the operator guesses a third time instead of stopping to ask. There is also no shared vocabulary of what "correct" looks like for nail shapes, finishes, and techniques.

---

## Goals

1. **Visual evaluation** — Claude evaluates every generated image after generation, before Rick sees it, and annotates failures with a one-line reason. Rick can correct wrong calls.
2. **Operator questioning** — when the same failure category appears in the two most recent low-rated outputs, the operator stops and asks a targeted question instead of guessing.
3. **Nail knowledge base** — a static document defining nail fundamentals (shapes, finishes, techniques), combined with a dynamic personal standards layer built from Rick's winners and corrections.

---

## Architecture

Three components work sequentially:

```
Generate images
  → Evaluate each image (knowledge base + personal standards)
  → Store annotations on output records
  → Display annotations in review UI (Rick can correct)
  → Rick rates images
  → Next run: pre-flight check — are the 2 most recent low-rated outputs both failing same category?
  → If yes: stop and ask targeted question
  → Otherwise: generate next batch
```

---

## Component 1: Nail Knowledge Base

### Static Layer (`operator/nailKnowledge.md`)

Written once, updated manually as nail standards evolve. Covers:

- **Shape definitions** — coffin, almond, stiletto, square, oval. Includes: proportions, tip shape, sidewall angle, what incorrect looks like for each.
- **Finish types** — cat eye (gradient/shimmer behavior, NOT harsh contrast), chrome (reflectivity level), French tip (placement, thickness), gel vs acrylic surface texture, glitter (density, coverage), foil behavior.
- **Technique criteria** — what constitutes a pass vs fail for each finish type.

Loaded into Claude's prompt on every evaluation call. Qualifies for prompt caching (rarely changes).

### Dynamic Layer (added to project brief)

Generated by the analysis engine from:
- Winner outputs (notes, what Rick approved)
- High-rated output notes
- Annotation corrections (where Rick marked Claude as wrong)

Stored as `brief.personalStandards` (plain text, line-separated). Updates automatically as Rick works. No manual maintenance.

---

## Component 2: Visual Evaluation

### When it runs

After Gemini returns images, before output records are saved to storage. Images are available in memory as `{ base64, mimeType }`.

### What it does

New module `operator/evaluate.js` calls Claude (Haiku — fast, vision-capable, cheap) with each image + the knowledge base. Returns for each image:

```json
{ "pass": true/false, "note": "one-line reason if failing, empty if passing", "category": "shape|finish|color|technique|other" }
```

Failures only: `pass: false` means something specific didn't match the spec. A passing image gets an empty note.

### Where results go

Stored on the output record as:

```json
"operatorAnnotation": { "pass": true/false, "note": "...", "category": "..." }
```

### Display in review UI

`ImageViewer.jsx` — a small label appears above the rating buttons when `operatorAnnotation.pass === false`:

> ⚠ Claude flagged: *nail tip too narrow for coffin shape* — [Correct]

The **Correct** button calls `POST /api/operator/annotation-correction` with `{ outputId, corrected: false }` (meaning Claude was wrong — the image passed). This correction is stored on the output record and fed back into the personal standards layer via the analysis engine.

If `operatorAnnotation.pass === true` (or no annotation), nothing is shown.

### Correction data shape

Added to output record:

```json
"annotationCorrection": { "corrected": false, "reason": "", "correctedAt": 1234567890 }
```

`corrected: false` means "Claude called a failure but Rick says it passed" — this is the only correction type in Phase 2.

## Component 3: Operator Questioning

### Trigger

Pre-flight check in `operator/run.js`, runs after loading outputs and before building the prompt.

Logic:
1. Find the two most recent outputs where rating ≤ 2 AND `operatorAnnotation.pass === false`
2. If both share the same `operatorAnnotation.category`, the same failure has appeared twice in low-rated outputs
3. Stop. Print a targeted question. Exit without generating.

### Targeted questions by category

| Category | Question |
|----------|----------|
| shape | "I've flagged nail shape twice in a row — can you send a reference image or describe the exact tip shape you want?" |
| finish | "The finish isn't landing after two tries — can you point to a winner that had it right, or describe what's missing?" |
| color | "The color isn't working after two tries — want to give me a hex code instead of a color name?" |
| technique | "The technique keeps coming out wrong — can you describe what correct looks like, or share a reference?" |
| other | "I've flagged the same issue twice and I'm guessing at this point — what specifically needs to change?" |

### What happens next

Rick responds verbally to the operator. The operator saves the answer via the existing `POST /api/operator/feedback` endpoint (notesKeep/notesFix), then continues to the next generation.

If Rick says "just keep going" — the operator clears the questioning flag and generates.

## What This Is Not

- Not a filter — no images are hidden from Rick
- Not a blocker — Rick's rating always wins
- Not a replacement for verbal feedback — it augments it
- Not automatic prompt rewriting — operator still writes the next prompt

---

## Out of Scope (Phase 3+)

- Winners automatically becoming visual refs
- Retroactive annotation pass on existing outputs
- Annotation history / trend visualization
- Corrections feeding into prompt generation directly
